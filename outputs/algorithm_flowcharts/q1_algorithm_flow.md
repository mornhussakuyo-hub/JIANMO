# Q1 算法流程说明

![Q1 算法流程图](q1_algorithm_flowchart.png)

当前 Q1 采用“ServiceUnit 压缩 + Giant Tour + Split DP + ALNS + 成本导向车辆复用排班”的启发式框架。程序首先读取 `customers.json`、`vehicles.json` 和 `distance_matrix.csv`，聚合正需求客户，展开具体车辆实例，并校验距离矩阵是否覆盖仓库与所有正需求客户。

在任务构造阶段，客户订单被压缩为求解器直接操作的 `ServiceUnit`。当前主要使用 `customer_sliced` 模式：同一客户需求可按目标重量和体积切片，超容量订单继续拆分，最终保证每个任务块都能被至少一种车型承运，同时保持总重量、总体积与原始订单守恒。

路线评价层由 `TrafficProfile`、`ArcCostCalculator` 和 `RouteEvaluator` 组成。评价器统一计算容量约束、分段时变行驶时间、能耗成本、碳成本、等待成本、迟到成本、服务时间和返仓时刻。后续初始解、Split DP、ALNS、车辆排班和最终校验都调用同一套评价逻辑，避免不同阶段口径不一致。

初始解阶段生成多类任务序列候选，包括难度排序、最近邻 Giant Tour、MST + DFS Giant Tour 和角度扫描 Giant Tour。Giant Tour 只给出任务的全局访问顺序，真正的路线切分由 Split DP 完成。Split DP 的状态转移为 `dp[j] = min_i dp[i] + route_cost(i,j)`，其中 `route_cost(i,j)` 表示将连续片段 `tour[i:j]` 作为一条路线时，在可行车型和发车时刻中取得的最低路线成本。

车辆允许真实复用后，固定启动车辆成本按唯一车辆实例计一次。当前 Q1 已将车辆复用排班改为成本导向：对每条路线枚举兼容车型、具体车辆实例和通用发车候选，要求同一车辆的多趟路线时间不重叠，并按“路线变动成本 + 新启动车辆固定成本”优先选择车辆与发车时刻。通用发车候选来自原始发车时刻、客户时间窗附近点、交通时段边界以及最晚时间窗后的缓冲候选。

初始候选解按照“未分配任务最少、总成本最低、使用车辆数更少、总距离更短”的规则择优。随后进入 ALNS 搜索：破坏算子包括随机移除、整路线移除、最差任务移除和可选相关路线移除；修复算子包括贪心插入和 regret-2 插入；搜索过程使用模拟退火接受准则与自适应权重，并在每个候选解上刷新真实车辆复用排班。

ALNS 结束后继续执行路线消除、路线内 2-opt、车型/车辆实例重分配和可选最终局部暴搜。路线消除尝试把整条路线的任务重新插回其他路线以压缩路线数；2-opt 改善路线内访问顺序；车型重分配只在排班可行且总成本改善时接受。最终解会经过 ServiceUnit 覆盖、容量守恒、路线评价一致性、车辆复用时间不重叠等校验，并输出 JSON、KPI、路线弧段、停靠点、客户服务、车辆使用和报告文件。
