# 问题1：静态环境下城市绿色物流配送调度方法

## 1. 问题理解与建模定位

问题1要求在无绿色配送区限行政策的静态环境下，为物流企业设计每日配送方案，使总配送成本最低。配送任务具有以下核心特征：

1. 配送中心编号为 `0`，客户编号为 `1` 到 `98`。
2. 附件给出任意两点间实际道路距离矩阵，因此路径长度不必由坐标欧氏距离估计。
3. 车辆为异构车队，包括 3 类燃油车和 2 类新能源车，车辆容量、容积、数量不同。
4. 每个客户有软时间窗。早到不算违反约束，但产生等待成本；晚到也允许，但产生惩罚成本。
5. 车辆行驶速度随出发时刻所在交通时段变化，行驶时间和能耗均受速度影响。
6. 能耗与速度呈 U 型关系，且车辆负载越高，单位距离能耗越高。

因此，问题1本质上是一个：

**带软时间窗、时变速度、负载相关能耗和异构车队容量约束的车辆路径问题**，可记为 Heterogeneous Fleet Vehicle Routing Problem with Soft Time Windows and Time-Dependent Load-Dependent Energy Cost。

传统 CVRP 或普通 VRPTW 只最小化车辆数或里程，不能直接覆盖本题。更合适的思路是把成本拆成启动成本、能源成本、碳排放成本、等待成本和迟到惩罚成本，并在路径评价函数中逐段计算。

## 2. 数据预处理与关键数据事实

清洗后的数据包括：

| 文件 | 含义 |
|---|---|
| `cleaned_data/distance_matrix.csv` | 配送中心与 98 个客户之间的道路距离矩阵 |
| `cleaned_data/customers.json` | 客户坐标、订单列表、时间窗、是否在绿色区域 |
| `cleaned_data/vehicles.json` | 5 类车辆参数 |

经清洗检查：

1. 距离矩阵为 `99 x 99`，含配送中心和 98 个客户。
2. 有需求订单的客户为 88 个，10 个客户在订单表中没有订单。
3. 总订单数为 2169 条。
4. 全部订单总重量约为 `285122.65 kg`，总体积约为 `772.43 m^3`。
5. 若直接按客户聚合需求，多个客户总需求超过任意一辆车容量。例如客户 55 总重约 `12197.65 kg`，客户 8 总体积约 `36.03 m^3`。
6. 存在 2 条单订单重量超过最大车辆载重 `3000 kg`：
   - 客户 31，订单 1686，重量约 `5987.05 kg`，体积约 `13.80 m^3`。
   - 客户 55，订单 1287，重量约 `3165.69 kg`，体积约 `8.70 m^3`。

这说明问题1不能简单建立“每个客户恰好访问一次”的模型，否则模型会不可行。更合理的处理是：

1. 客户允许被多辆车多次服务。
2. 订单原则上可合并配送。
3. 对超过最大单车容量的订单，视作可拆分订单，拆成若干同客户、同时间窗、同位置的配送子任务。

本文后续采用“配送任务包”作为建模对象。任务包可以理解为同一客户的一批订单或订单片段，满足最大车辆容量上限。这样既保留客户多次配送的现实可能，也避免直接对 2169 个订单建路径模型导致规模过大。

## 3. 基本假设

为使模型可计算并与题面信息一致，作如下假设。

### 3.1 配送与服务假设

1. 所有车辆均从配送中心出发，并最终回到配送中心。
2. 每辆车一天内最多执行一条配送路线。如需建模多趟配送，可把同一物理车辆的多趟视作多个可用车辆资源；问题1中车辆数量充足，可先采用单路线假设。
3. 同一客户可被多辆车服务，也可被同一辆车在一条路线中服务一个任务包。
4. 每次到达客户点的服务时间为 `20 min`。
5. 对同一客户的多个任务包，如果被安排在同一条路线且连续服务，可以合并为一次客户访问，只计一次服务时间；算法实现中通过“同客户任务包合并”局部优化处理。

### 3.2 时间窗假设

设客户 `i` 的最早到达时间为 `e_i`，最晚到达时间为 `l_i`。

1. 若车辆到达时间 `A_i < e_i`，车辆等待到 `e_i` 后开始服务，等待成本为 `20 元/小时`。
2. 若车辆到达时间 `A_i > l_i`，允许服务，但产生迟到惩罚成本 `50 元/小时`。
3. 若 `e_i <= A_i <= l_i`，不产生时间窗成本。
4. 每个客户的所有订单沿用同一客户时间窗。

### 3.3 速度时变假设

题面给出 8:00 到 17:00 的交通状态。由于部分时间窗延伸到晚上 20:37，为保持全天路线可评价，建议作如下扩展：

1. 8:00-9:00、11:30-13:00 为拥堵时段。
2. 10:00-11:30、15:00-17:00 为一般时段。
3. 9:00-10:00、13:00-15:00 为顺畅时段。
4. 17:00 之后缺少题面速度信息，主模型中按顺畅时段处理；论文中可在敏感性分析部分比较“17:00 后按一般时段”的结果。

题面速度服从正态分布。问题1的主优化模型建议先采用确定性等价形式，即用各时段期望速度计算行驶时间和能耗：

| 时段类型 | 期望速度 km/h |
|---|---:|
| 顺畅 | 55.3 |
| 一般 | 35.4 |
| 拥堵 | 9.8 |

随机速度可作为后验仿真验证：固定调度方案后，抽样速度分布，评估总成本均值、方差、迟到风险。

### 3.4 能耗和负载假设

车辆基础能耗只由速度决定，满载能耗较空载上升。对路径中每一段弧 `(i,j)`，以车辆离开点 `i` 时的剩余载重计算负载率：

```text
rho = remaining_weight / vehicle_max_weight
```

其中 `rho` 截断在 `[0,1]` 内。

燃油车满载比空载能耗高 40%，新能源车满载比空载能耗高 35%。因此：

```text
fuel_load_factor = 1 + 0.40 * rho
electric_load_factor = 1 + 0.35 * rho
```

题面未给出体积对能耗的直接影响，因此体积只用于容量可行性约束，不进入能耗函数。

### 4.3 路线变量

直接建立弧变量模型会因同客户多任务包、异构车辆、时变速度和软时间窗而较复杂。更推荐使用路线集合模型：

```text
y_r = 1  表示选择路线 r
y_r = 0  表示不选择路线 r
```

其中路线 `r` 已经绑定某一车型 `k(r)`，并包含一串任务包访问顺序：

```text
r = (0, s_1, s_2, ..., s_m, 0)
```

若任务包 `s` 被路线 `r` 服务，记：

```text
a_sr = 1
```

否则 `a_sr = 0`。

## 5. 配送任务包构造

### 5.1 为什么不直接以客户为节点

若把客户总需求直接聚合成一个需求点，要求每个客户恰好访问一次，则必须满足：

```text
sum_orders_weight_i <= max_k W_k
sum_orders_volume_i <= max_k Q_k
```

但清洗数据中多个客户明显超过最大车辆容量，因此该模型不可行。

### 5.2 任务包构造原则

对每个客户 `i`：

1. 收集其所有订单。
2. 若某个订单重量或体积超过最大车辆容量，则先拆成若干订单片段。
3. 使用一维/二维装箱思想，把同客户订单合并成若干任务包。
4. 每个任务包必须满足：

```text
w_s <= max_k W_k
q_s <= max_k Q_k
```

本题中最大车辆能力为：

```text
max_k W_k = 3000 kg
max_k Q_k = 15 m^3
```

具体可采用 First-Fit Decreasing 算法：

1. 对客户 `i` 的订单按 `max(weight / 3000, volume / 15)` 从大到小排序。
2. 依次尝试放入已有任务包。
3. 若加入后不超过 `3000 kg` 和 `15 m^3`，则合并。
4. 否则新建任务包。

这样构造出的任务包是路线模型的基本服务对象。同一客户可能产生多个任务包，代表该客户允许被多车多次配送。

### 5.3 可进一步优化的任务包拆分

仅用最大车辆容量装箱可能导致部分任务包只能由大型车服务。为了提升车辆匹配灵活性，可在算法中动态调整任务包：

1. 若某任务包重量 `> 1500 kg` 或体积 `> 10.8 m^3`，则不能由中小燃油车服务。
2. 若新能源大车数量紧张，可把部分大任务包进一步拆分，让燃油小车承担。
3. 若某条路线中同客户两个任务包连续出现，可合并访问，减少服务时间。

这类拆分与合并属于启发式求解中的局部优化动作。

## 6. 时变行驶时间计算

车辆从节点 `i` 到节点 `j` 的距离为 `d_ij`。由于速度随时段变化，不能简单用：

```text
travel_time = d_ij / average_speed
```

而应根据车辆离开节点 `i` 的时间逐段推进。

设车辆在时刻 `t` 离开节点 `i`，还需行驶距离 `D = d_ij`。算法如下：

```text
function TimeDependentTravel(i, j, t):
    D = d_ij
    current_time = t
    total_time = 0
    while D > 0:
        period = traffic_period(current_time)
        v = expected_speed(period)
        period_end = end_time(period, current_time)
        available_time = period_end - current_time
        max_distance = v * available_time
        if D <= max_distance:
            delta = D / v
            total_time += delta
            current_time += delta
            D = 0
        else:
            D -= max_distance
            total_time += available_time
            current_time = period_end
    return total_time
```

该算法自动处理跨时段情况。例如一段路从 11:20 出发，先在一般时段行驶 10 分钟，11:30 后进入拥堵时段，再按拥堵速度继续计算。

为保持路径评价满足先进先出性质，速度函数只依赖实际出发时刻，不允许车辆通过人为推迟出发获得不合理的“超车收益”。若等待至更顺畅时段确实能降低总成本，模型可通过客户早到等待或配送中心延迟出发体现。

## 7. 能耗、碳排放与运输成本

### 7.1 基础能耗函数

题面给出燃油车每百公里油耗：

```text
FPK(v) = 0.0025 v^2 - 0.2554 v + 31.75
```

其中 `v` 单位为 km/h，`FPK(v)` 单位为 L/100km。

新能源车每百公里电耗：

```text
EPK(v) = 0.001 v^2 - 0.136 v + 6.194
```

其中 `EPK(v)` 单位为 kWh/100km。

速度取该段行驶所处交通时段的期望速度。若一段弧跨多个交通时段，则应拆成若干子段分别计算能耗。

### 7.2 负载修正

设车辆类型为 `k`，离开节点 `i` 时车辆剩余载重为 `L_i`，载重率为：

```text
rho_i = L_i / W_k
```

对燃油车：

```text
fuel_ij = distance_segment / 100 * FPK(v) * (1 + 0.40 rho_i)
```

对新能源车：

```text
electric_ij = distance_segment / 100 * EPK(v) * (1 + 0.35 rho_i)
```

这里的 `distance_segment` 是某一交通时段内实际行驶的距离。如果一段弧跨多个时段，则对每个子段求和。

### 7.3 能源费用

燃油价格为 `7.61 元/L`：

```text
energy_cost_ij = fuel_ij * 7.61
```

电价为 `1.64 元/kWh`：

```text
energy_cost_ij = electric_ij * 1.64
```

### 7.4 碳排放成本

燃油碳排放系数为 `2.547 kg/L`，新能源电耗碳排放系数为 `0.501 kg/kWh`，碳成本为 `0.65 元/kg`。

燃油车：

```text
carbon_cost_ij = fuel_ij * 2.547 * 0.65
```

新能源车：

```text
carbon_cost_ij = electric_ij * 0.501 * 0.65
```

### 7.5 弧运输成本

对弧 `(i,j)`：

```text
arc_cost_ij = energy_cost_ij + carbon_cost_ij
```

由于能耗受速度和负载共同影响，`arc_cost_ij` 不是固定常数，而依赖：

1. 车型。
2. 出发时刻。
3. 离开节点时的剩余载重。
4. 是否跨越多个交通时段。

## 8. 时间窗成本

设车辆到达任务包 `s` 所属客户的时刻为 `A_s`，客户时间窗为 `[e_s, l_s]`。

等待时间：

```text
wait_s = max(0, e_s - A_s)
```

迟到时间：

```text
late_s = max(0, A_s - l_s)
```

时间窗成本：

```text
tw_cost_s = 20 * wait_s + 50 * late_s
```

若早到，则服务开始时间为：

```text
B_s = e_s
```

否则：

```text
B_s = A_s
```

服务完成时间：

```text
D_s = B_s + 1/3
```

其中时间单位为小时。

## 9. 路线成本计算

考虑一条由车型 `k` 执行的路线：

```text
r = (0, s_1, s_2, ..., s_m, 0)
```

初始装载量：

```text
L_0 = sum_{u=1}^m w_{s_u}
U_0 = sum_{u=1}^m q_{s_u}
```

容量可行性要求：

```text
L_0 <= W_k
U_0 <= Q_k
```

设车辆从配送中心出发时刻为 `T_0`，可设为 8:00，也可作为路线局部优化变量。依次访问任务包时，执行以下递推：

1. 根据当前节点、下一客户、当前时刻、车型和剩余载重，计算行驶时间与运输成本。
2. 得到到达时刻 `A_s`。
3. 计算等待/迟到成本。
4. 加上服务时间 `1/3` 小时。
5. 卸货后更新剩余载重和剩余容积。

路线总成本为：

```text
C_r = F_k
    + sum_{(i,j) in r} arc_cost_ij
    + sum_{s in r} tw_cost_s
```

其中 `F_k = 400`。

路线返回配送中心的弧也应计入行驶成本和碳排放成本，但不再产生客户时间窗成本。

## 10. 路线集合优化模型

构造所有可能路线后，可建立集合划分模型：

```text
minimize   sum_{k in K} sum_{r in R_k} C_r y_r
```

满足：

### 10.1 每个任务包恰好被服务一次

```text
sum_{k in K} sum_{r in R_k} a_sr y_r = 1,  for all s in S
```

### 10.2 车型车辆数量约束

```text
sum_{r in R_k} y_r <= N_k,  for all k in K
```

### 10.3 路线选择变量

```text
y_r in {0,1}
```

每条路线 `r` 在进入模型前已经满足：

1. 载重容量约束。
2. 容积容量约束。
3. 时序递推可行。
4. 所用车型可执行该路线。

软时间窗不作为硬约束，而是进入 `C_r`。

该路线集合模型表达清晰，但所有可行路线数量巨大，不能枚举。因此实际求解时应采用启发式或列生成思想生成高质量路线池。

## 11. 求解算法设计

本题规模为 98 个客户、2169 条订单、5 类车型。直接精确求解完整混合整数规划难度较大，更适合采用：

**任务包构造 + 初始解生成 + 自适应大邻域搜索 ALNS + 车型重分配 + 路线池精修**。

整体流程如下：

```text
Step 1  读取清洗数据
Step 2  构造配送任务包
Step 3  生成初始可行解
Step 4  用 ALNS 反复破坏和修复路线
Step 5  对每条路线做局部搜索
Step 6  做车型替换和出发时间微调
Step 7  从搜索过程中保留的路线池中做集合划分精修
Step 8  输出车辆使用方案、路径、到达时间和成本构成
```

### 11.1 初始解生成

初始解需要快速构造可行路线。建议采用“时间窗优先的顺序插入法”。

1. 将任务包按最晚到达时间 `l_s` 从早到晚排序。
2. 依次尝试把任务包插入已有路线的任意位置。
3. 对每个候选插入位置，完整重新计算路线成本。
4. 选择增量成本最小且容量可行的位置。
5. 若无法插入任何现有路线，则开启一辆新车。
6. 新车优先选择单位容量综合成本更低的车型，并兼顾车辆数量限制。

插入增量成本定义为：

```text
Delta = C_route_after - C_route_before
```

由于路径成本具有时变性和负载相关性，不能只用距离增量近似，必须完整重评估路线。

### 11.2 车型初选规则

开新路线时可采用以下经验规则：

1. 若任务包重量或体积超过中小车容量，只能选择大车。
2. 对同等可行路线，优先比较完整路线成本，而不是固定优先使用新能源车或燃油车。
3. 新能源车单位能源成本和碳成本通常较低，但数量少，应优先用于：
   - 距离较长的路线。
   - 总载重较大的路线。
   - 时间窗较宽、可稳定完成的路线。
4. 燃油大车数量较多，适合作为主要承运能力。

### 11.3 ALNS 框架

ALNS 每轮从当前解中移除一部分任务包，再用修复算子重新插入。其优势是能跳出普通局部搜索的局部最优。

伪代码：

```text
Initialize solution x
best = x
temperature = T0

while stopping criterion not met:
    choose destroy operator d by adaptive probability
    choose repair operator r by adaptive probability
    removed = d(x)
    x_new = r(x without removed, removed)
    x_new = local_search(x_new)
    x_new = vehicle_reassignment(x_new)

    if accept(x_new, x, temperature):
        x = x_new

    if cost(x_new) < cost(best):
        best = x_new

    update operator scores
    cool down temperature

return best
```

接受准则可采用模拟退火：

```text
accept probability = exp(-(cost(x_new) - cost(x)) / temperature)
```

若新解更优，则必然接受。

### 11.4 破坏算子

建议设计多个破坏算子，并由 ALNS 自适应选择。

#### 随机移除

随机移除一定比例的任务包，增强搜索多样性。

#### 相关性移除

选择一个种子任务包 `s`，移除与其相似的一批任务包。相似性可定义为：

```text
related(s,u) =
    alpha * distance(g(s), g(u))
  + beta  * |e_s - e_u|
  + gamma * |l_s - l_u|
```

相关任务包被重新插入后，容易形成更紧凑的路线。

#### 最差贡献移除

计算每个任务包从当前路线移除后能节省的成本，优先移除贡献成本高的任务包。

```text
saving_s = C_route_before - C_route_after_removing_s
```

这有助于修复路径中位置很差的任务包。

#### 时间窗冲突移除

优先移除迟到惩罚较高或等待成本较高的任务包，重新安排其服务顺序。

#### 车型冲突移除

选择成本较高的某类车辆路线，将其中任务包释放出来，尝试由其他车型重新组合。

### 11.5 修复算子

#### 贪婪插入

每次选择一个任务包，插入使总成本增量最小的位置。

#### Regret-2 插入

对每个未插入任务包，计算最佳插入成本和次佳插入成本：

```text
regret_2 = second_best_delta - best_delta
```

优先插入 `regret_2` 最大的任务包。含义是：如果现在不插入它，之后可能付出更大代价。

#### Regret-3 插入

同理比较前三个插入位置，适合任务包数量较大时使用。

#### 车型感知插入

插入时不仅比较位置，也比较车型替换可能。例如一条燃油大车路线若总载重和体积降低后可由新能源小车执行，则同步评估车型替换后的路线成本。

### 11.6 局部搜索算子

每次修复后，对路线进行局部优化。

#### 路线内 2-opt

反转同一路线中的一段访问顺序，降低距离和时间窗成本。

#### 跨路线 relocate

把一个任务包从路线 A 移动到路线 B 的某个位置。

#### 跨路线 swap

交换两条路线中的任务包。

#### cross-exchange

交换两条路线中的连续任务包片段。

#### 同客户任务包合并/分离

若同一客户的多个任务包分散在多条路线，可尝试合并到一条路线；若合并后容量超限或造成严重迟到，则保持分离。

#### 出发时间微调

每条路线的配送中心出发时间不一定固定为 8:00。对一条路线，在不增加迟到太多的情况下，延后出发可能减少早到等待成本；在拥堵前提前出发可能降低运输成本。可对路线出发时间做小范围枚举，例如：

```text
T0 in {8:00, 8:10, 8:20, ..., 10:00}
```

选择路线成本最低的出发时间。

## 12. 路线成本重评估细节

每次插入、删除、交换任务包后，都必须重新计算整条路线，而不是只更新局部距离。原因是：

1. 访问顺序改变会改变每个节点的到达时间。
2. 到达时间改变会影响交通时段，从而影响行驶速度。
3. 服务顺序改变会改变每条弧上的剩余载重，从而影响能耗。
4. 时间窗成本是非线性的分段函数。

路线重评估函数应输出：

```text
{
  "route": [0, ...customers..., 0],
  "vehicle_type": k,
  "departure_time": T0,
  "arrival_times": {...},
  "service_start_times": {...},
  "waiting_cost": ...,
  "late_cost": ...,
  "energy_cost": ...,
  "carbon_cost": ...,
  "startup_cost": 400,
  "total_cost": ...
}
```

这正好对应题目要求的车辆使用方案、行驶路径、到达时间及成本构成。

## 13. 成本目标函数展开

最终总成本可写为：

```text
TotalCost =
    StartupCost
  + EnergyCost
  + CarbonCost
  + WaitingCost
  + LatePenalty
```

其中：

```text
StartupCost = sum_used_vehicles 400
```

```text
WaitingCost = sum_customers 20 * max(0, e_i - A_i)
```

```text
LatePenalty = sum_customers 50 * max(0, A_i - l_i)
```

能源和碳排放成本按所有车辆、所有行驶弧、所有跨时段子段累加。

若路线集合模型中已预先计算每条路线成本，则：

```text
TotalCost = sum_r C_r y_r
```

## 15. 模型可行性与异常数据处理

### 15.1 零订单客户

订单表中没有订单的客户不需要被服务。它们仍可保留在距离矩阵和客户数据中，但不进入任务包集合 `S`。

### 15.2 超最大容量订单

单订单超过最大载重时，若订单不可拆分，问题无可行解。本题作为城市电商配送场景，更合理的处理是允许超大订单拆成多件或多批次配送。因此对超容量订单拆分为多个子任务。

例如订单重量 `5987.05 kg`，可拆为：

```text
ceil(5987.05 / 3000) = 2
```

个子任务，每个子任务体积按重量比例分摊。

### 15.3 缺失数据

若某任务包缺失时间窗，可使用宽时间窗占位，例如：

```text
[8:00, 21:00]
```

但应在报告中标记。当前清洗数据未发现明显缺失。

## 16. 随机速度的后验检验

主优化采用期望速度得到确定性方案。为体现题面速度随机性，建议在得到最优或近优方案后做 Monte Carlo 检验。

步骤如下：

1. 固定车辆路径和出发时间。
2. 对每个交通时段按题面正态分布抽样速度。
3. 速度若小于安全阈值，例如 `5 km/h`，则截断为 `5 km/h`。
4. 重新计算所有路线到达时间、等待成本、迟到成本、能耗和碳排放。
5. 重复 500 到 1000 次。
6. 输出总成本均值、标准差、95% 分位数和迟到概率。

这样可以说明方案不仅在平均速度下成本低，而且对交通波动具有一定稳健性。

在论文中，问题1可以按以下逻辑展开：

## 19. 方法小结

问题1的关键不是简单找最短路，而是处理三类耦合关系：

1. **容量耦合**：车辆载重、容积和客户多订单需求共同决定能否合并配送。
2. **时间耦合**：访问顺序改变到达时间，到达时间改变交通状态，交通状态又改变后续行驶时间。
3. **成本耦合**：速度影响能耗，负载影响能耗，车型影响能源价格和碳排放系数。

因此推荐方法是：先把订单整理成容量可行的任务包，再用能精确评价时变路线成本的 ALNS 搜索车辆路径，并在最终阶段输出完整的车辆、路径、时间和成本明细。该方法既能满足题面数学建模要求，也能落地为可运行求解程序。
